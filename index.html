Rust Web Wasm

Summary

This project is a Turing Machine time complexity analyser. Written in Rust using Wasm, Yew, and DaisyUI, this web-based project implements novel functionality to aid in exploring the composition of Turing Machines.

Information

This is a web-based project built in Rust, designed to analyse the time complexity of a given Turing Machine, returning a Big-O notation estimation from empirical dynamic runtime analysis utilising genetic algorithm-based and a uniquely extended grammar-based fuzzing approach.

This project is available here

Concept and Features

Turing Machines are an abstract model of computation, popular in automata theory for their simplicity and computational equivalence to modern computers. This popularity has brought a significant number of Turing Machine simulators, designed to assist students in understanding their workings and importance. Broadly, existing simulators implement minimal functionality, missing various features a student may find useful in aiding understanding. This project implements one such feature, through bridging the comprehension gap between Turing Machines and contemporary computer programs through implementing time complexity estimation. This serves to both guide the production of such machines alongside providing a metric to liken their execution to prior taught algorithms.

This is achieved through the use dynamic runtime analysis, through two methods: genetic algorithms and grammar-based fuzzing. These dynamically produce inputs for the Turing Machine over various input sizes, which are subsequently executed to produce a list of inputs and their respective runtimes, allowing for estimation of runtime. Uniquely, the approach used for grammar-based fuzzing implements variables within repetition ranges, integrating context awareness, increasing its capabilities beyond those of standard fuzzers. This allows much better exploration of Turing Machines through partially bypassing the limitations of prior solutions.

These results are then provided to the Wasm/Yew Rust frontend and plotted using JavaScript, resulting in clear and concise graphs breaking down runtimes, estimations, and state visits for all states over all runtimes. This provides the information required to better allow users to understand Turing Machines.

Challenges

This project implements a multi-level lexer, including support for both input grammars and mathematical expressions within contained ranges, alongside variables within ranges. The greatest challenge faced was the implementation of variable-based grammatical fuzzing, requiring extensive research into relevant algorithms and implementation planning. This was a complex challenge, requiring a combination of previously unknown algorithms to reach a working and highly performant solution.

Additionally, this project was built on existing work, necessitating reading documentation and understanding the previously built solution and component interaction to meaningfully proceed. This was the first major project constructed in this way, and taught the necessity of producing and reading high-quality documentation to understand prior work.

Summary

This project is a convolutional neural network-based image classifier. Written in Python using PyTorch, this project fine-tunes and prunes the MobileNet V3 classification model for use on a curated set of images.

Information

This is a Jupyter-based project built in Python, designed for the fine-tuning, pruning, and evaluation of a custom dataset of images of flowers. This project modifies the MobileNet V3 architecture in a full end-to-end classification pipeline.

Concept and Features

This project involved collecting and processing images of various flowers, involving capturing images across various conditions, backgrounds, and angles to reduce overfitting in the resulting model. These images were divided into training and test datasets, cropped, and labelled. Further, the training dataset used in the project itself implements further augmentations to these images, implementing a resize to fit the model input specification, random rigid transformations, and colour jitter to reduce model overfitting.

Based on this data, this project then fine-tunes an existing model (MobileNet V3) using the Cross Entropy Loss function with the Adam optimiser to achieve high accuracy on the dataset over a small number of epochs with a decaying learning rate. After, this model is used in pruning, wherein each conducted prune removes a small amount of the model (~6%), before conducting fine-tuning to recover accuracy. This process is repeated until the desired portion of parameters are removed.

This results in a significantly smaller yet still highly accurate model, capable of performing extremely similarly in the provided test dataset to the prior full model. In testing, this resulted in recorded accuracies of 98-100% pre-pruning, and 97-100% post-pruning while model parameters and FLOPs were reduced 55%.

Challenges

This project achieves a result of reducing the model size by 55%. MobileNet V3 is already a compact architecture relative to contemporary architectures like DenseNet, and thus is prone to brittleness. This required an iterative pruning-healing approach to ensure that the network was not permanently damaged by overpruning. Tuning hyperparameters and implementing this approach constituted the largest challenges faced in the implementation of this project.

Summary

This project was written in Java using the JOGL library, a library which aims to import OpenGL functionality to Java. It is a basic animated scene which uses low-level graphical techniques and shaders.

Information

This is a graphical project built using the JOGL library for Java, alongside JSwing to handle UI elements. This was built over the course of 2 weeks as coursework for a university module.

Concept and Features

This project is an animated scene built using low-level graphics concepts. This consists of an animated bee, statues (which feature proximity detection), and a backdrop, made from billboarding trees and a sun. Each statue in the scene has a different affect on the bee, as shown in the demo - the wooden statue makes the bee flip over, the stone statue makes the bee circle around it, and the metal statue increases the flow of time. When a bee is in proximity of a statue, its face will change to reflect how long it has been near the bee.

Objects in the scene and their animations (which are mostly determined by simple sine wave functions) are handled using transforms applied to nodes in a scene graph, and rendered by the GPU using vertex and fragment shaders. The fragment shader supports multiple lights, multiple types of light (pictured are directional and spotlights, but point lights are also supported), and has partial support for transparency. It uses the Blinn-Phong reflection model to shade objects and supports light attenuation for point lights and spotlights.

Challenges

This project represents an introduction to low-level graphics programming. The primary challenge of this project was expectedly learning shader code - shaders are highly parallel and require a different methodology behind their composition, as data is not directly transferrable between shader iterations. Instead, data is either provided by the previous shader or provided by the CPU, passed in as a uniform. Both of these methods work very differently to how typical CPU code handles data, and requires extensive learning and planning to operate effectively.

Alongside this, modelling in this project was done without the use of previously used tools like Blender, as was used before in the Alchemy Game. Instead, models were designed through a scene graph, using nodes to produce primitive shapes and the transforms which separated, scaled, and animated them.

Summary

This project was written in Python using NumPy and SciPy. It is a classifier designed to classify images of numbers in the MNIST dataset.

Information

This project was created over a week, and is a classifier model built to identify the number represented by an image in the MNIST dataset.

Over the production of this project, I learned a large number of statistical techniques for classification, such as LDA, PCA, KL divergence, and k-NN classification.

Concept and Features

This classifier was built for the MNIST database, a popular database for training and testing machine learning and classification systems due to its relatively large size (60,000 training images, 10,000 testing images) and high data quality. It uses 28x28 greyscale images of numbers 0-9. I used 5,000 of these images with my classifier to increase speed.

My classifier was built to specifically identify noisy and masked MNIST data, and so features extensive dimensionality reduction - a technique used to simplify high-dimensional data, which in my case, after extensive parameter tuning, reduced the 784 parameters (28x28) down to 62 parameters. Because fewer, more important parameters are used, the system is much more resilient to noise, as noise typically has the greatest impact on empty pixels - which are usually unused as they are usually unimportant and portray little data, causing dimensionality reduction to discard them.

With my reduced parameters, I was able to achieve 95% accuracy on noisy data and 79% accuracy on masked data, which are relatively good results considering the reduced dataset that this model was trained on.

Challenges

This project necessitated learning a large amount of the mathematics used in image classification and translating it as efficiently as possible to Python. To help in handling large array, vector, and matrix operations, I learned the basics of NumPy, an efficient C-based Python library. Using the broadcasting feature of NumPy arrays allows functions to be quickly applied to all elements of an array, allowing for the model to be lazy-trained and classify 1,000 testing images within 5 seconds of it being run.

Additionally, there are many possible techniques for dimensionality reduction and classification, and it is hard to know which one is perfect for a given application. Initially, LDA - a technique used to reduce the dimensions of a dataset to a few dimensions which provide the most distinction between options - was used in conjunction with PCA, however this was found to reduce performance significantly, especially on masked data, and so was removed, leaving only PCA and KL divergence for dimensionality reduction. This amongst other techniques had to be gradually tweaked, such as classifier originally using Euclidean distance instead of cosine distance, which was also found to massively reduce performance on masked data.

Summary

This project was written in pure Haskell. It is a CLI-based game designed to simulate games of Texas Hold Em' poker between both bots and humans.

Information

This project was created over around three weeks, and is a recreation of Texas Hold Em' poker, complete with support for AI bots and Jokers.

This project was produced using an entirely new programming paradigm - functional programming. Haskell is a purely functional language and thus works very differently from the procedural and object-oriented programming languages previously used.

Concept and Features

This game is a recreation of Texas Hold Em' poker, a popular card game.

This version allows for human players to play alongside bots, which use a variety of basic strategies - of which there are 4 implemented.
  • A random AI, which makes moves entirely based on chance without considering the game state.
  • A passive AI, an extension of the random AI with more passive tunings, which moves based on the current bet amount.
  • An aggressive AI, an extension of the random AI with more aggressive tunings, which moves based on its chip advantage.
  • A smart AI, which uses a complex function to evaluate its hand and how it should fare relatively, alongside a fuzzy logic-based decision tree to decide whether a bet is worth taking.
Additionally, Jokers were implemented as wildcards. Any number can be added to the deck, and will always produce the best possible result.

Challenges

This project necessitated learning an entierely new programming paradigm. Haskell, amongst many functional programming languages enforces a few strange practises, the most striking of which being the inherent limitation on the length of a function. The lack of a true for or while loop in the language forces a recursive solution to most problems, necessitating restructuring logic.

Other major complications included implementing an initially unplanned feature - Jokers. The addition of Jokers required a rewrite of all functions related to checking hand strength and getting a certain hand type. These rewrites typically significantly increased complexity as the logic to always get a Joker in the correct place is sometimes quite complex. One notably complex example was checking for straights, which Jokers must always both attempt to complete and find the best available straight.

Finally, the last major challenge was debugging. Haskell is a pure functional programming language and evaluates expressions lazily; values are not calculated until they are required elsewhere in the program. This can greatly increase the difficulty of debugging, as the debug trace will not be evaluated unless the expression itself is, making debug behaviour somewhat unpredictable through causing infections to propagate unexpectedly far away.

Summary

This project was written in Java using JSwing for the GUI. It is a graphical application designed for querying a database.

Information

This project was created in around two weeks, produced in Java using the JSwing library. This project implements an SQL-like syntax for searching a given database, through exploring the fundamental concepts behind interpreted languages. This involved producing a basic lexer in Java to parse and combine queries using logical operators.

Concept and Features

This project formats and searches a database. To do this, an SQL-like query language was written supporting the use of various tokens. These include mathematical operators (<, <=, ==, >, >=), comparative string operators (begins, contains, equals), and logical operators (AND, OR, XOR, NAND, NOR, XNOR) and brackets to connect subqueries, alongside the NOT operator to invert a subquery.

To format the database, the database is scanned as it is read and data types are categorised as either numeric or a string (based on whether entries can be parsed as a float), and either filterable or non-filterable (based on how many unique entries there are). This is used to generate filters and choose the correct type of operator for the input given.

The language also supports the selection of any number of valid database data types in its arguments alongside the use of the wildcard (*) to select all data types. These can be chained in any order.

Alongside this, I built a class to combine any two given databases by a common type - the demo for this project uses a modified version of the publically availible IMDB database, having combined the ratings and basic databases.

Challenges

The most significant challenge of producing this project was in UI design. JSwing requires a heavily nested approach to UI design, which through the verbosity of Java can quickly become complex and obfuscate prior UI component function. This required a better approach to code structure and a modification to naming conventions to reduce variable ambiguity, thereby teaching their importance.

The other major challenges was in producing the query parser and the necessary classes to flexibly and dynamically support query parsing. Setting up these classes was relatively complex compared to many of the projects previously completed, and represented the first time modularity was heavily considered as a design component.

Summary

This project was written in C# using the Unity game engine. It is a 3D game about time management and cooking.

Information

This project was created over the span of two weeks as part of a "game jam" - a time-limited development event wherein participants are required to to produce a video game following a provided theme. Participants may enter alone or as a group; this project was built with one other developer.

This project is heavily graphics focused by design, necessitating the use of graphical design software. To this end, Blender was used to produce basic untextured 3D models for use as game assets, using the Unity game engine to provide implementations for rendering and physics. Written in C#, this project implements a cooking game based on receiving and fulfilling orders through producing potions.

Concept and Features

In this game, players must rush to complete increasingly difficult procedurally generated orders of potions as fast as possible, by processing ingredients and adding them to a cauldron.

To allow players to do this, they are able to move around a 3D space and pick up ingredients to be processed or used in a cauldron.

To generate orders, potions are created with given recipes. Difficulty and time given are then dynamically assigned based on a number of factors. If the difficulty of a given potion fits within a certain threshold, the potion is eligible to be chosen as the next order.

To make the cauldron functional, all potions are represented as a subset of a central volume. To make a potion, ingredients - which function as vectors, must be added to move the cauldron's position to reach a position within the desired potion volume. To aid in this, players are given recipes corresponding to this position but are able to find their own, more efficient recipes.

Challenges

The main challenge associated with this project was precise co-ordination. Constant communication was required to develop features under intense time pressure while simulaneously learning 3D modelling and planning systems. Co-ordination was handled through scheduled meetings and burndown charts to track remaining features. This ensured that important features were given higher priority such that a functional prototype could be designed within the two week time limit.

This approach was largely successful, resulting in a functional and playable prototype designed within two weeks, through careful management of scope.

Summary

This project was written in C# using the Unity game engine. It is a 2D physics-based game about golf in space.

Information

This project was built over 3 months as the coursework submission for the Computer Science A-Level Coursework. This project was developed in the Unity game engine using C#.

Concept and Features

In this game, players must hit a golf ball to a flag as fast as possible and in as few swings as possible, utilising gravity to do so.

To aid a user in achieving this, they are provided a swing, executed through clicking and dragging on the screen, with a greater displacement magnitude corresponding to a higher power swing. Additionally, users are able to correct their trajectory using an additional mid-air velocity boost to improve ease of use.

The game features 7 levels, allowing users to play in singleplayer or turn-based local multiplayer.

Challenges

This project was produced rapidly and without extensive planning, iteratively adding features as they were required. This led to the project quickly accruing extensive technical debt as components were naively appended to the project without consideration of future features. This led to a highly coupled prototype which proved extremely brittle to modification, highlighting the necessity of a modular approach. Subsequent rewriting and refactoring of prior code taught me the importance of planning and code modularity.

Additionally, being the first highly visual project worked on, this required exploration of UI/UX design. Without a background or prior interest in graphical design, this proved difficult to perform eloquently. Despite this, this served as an invaluable introduction to design and inspired changes future approaches to interface production.
