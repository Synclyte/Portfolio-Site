Summary

This project was written in C# using the Unity game engine. It is a 2D physics-based game about golf in space.

Information

This project was built over the course of 3 months for the final assignment as a part of my A level Computer Science course.

To create this project, I had to learn C#, as this is what Unity (the game engine which I used) requires.

Concept and Features

In this game, players must hit a golf ball to a flag as fast as possible and in as few swings as possible, utilising gravity to do so.

To allow players to do this, they are given a variably powered swing alongside an omnidirectional double jump for minor trajectory correction. The game features 7 levels of varying difficulty - 5 of which can be played multiplayer, each showcasing different mechanics to attempt to make gravity a much more intuitive concept.

Challenges

With this being my first large project, I went into this with my prior mindset of very quickly producing a prototype, adding more functionality onto the prototype rapidly and then being done. This eventually produced an incomprehensible buggy mess as the codebase expanded, forcing me to refactor huge chunks of the code and self-teach the fundamentals of Software Engineering.

This was also the first time I had to produce a real user interface - most of my projects before this were just prototypes, and they were designed to only be used by me. While the end product is basic, this was a great learning experience.

C# Game Software Engineering Graphical

Summary

This project was written in C# using the Unity game engine. It is a 3D game about time management and cooking.

Information

This was my second major project. It was created over the span of two weeks, initially created for a game jam - an event where participants are challenged to make a working, polished game fitting a theme in a short amount of time - individually, or with a team.

To create this project, I had to learn how to 3D model, necessitating me learning Blender. Additionally, I produced this project with a friend, thus I had to learn how to organise development more efficiently.

Concept and Features

In this game, players must rush to complete increasingly difficult procedurally generated orders of potions as fast as possible, by processing ingredients and adding them to a cauldron.

To allow players to do this, they are able to move around a 3D space and pick up ingredients to be processed or used in a cauldron.

To generate orders, potions are created with given recipes. Difficulty and time given are then dynamically assigned based on a number of factors. If the difficulty of a given potion fits within a certain threshold, the potion is eligible to be chosen as the next order.

To make the cauldron functional, all potions are represented as volume masses in a 3D space. To make a potion, ingredients - which function as vectors, must be added to move the cauldron's position to the position within the potion's volume in 3D space. Players are given recipes corresponding to this position but are able to find their own, more efficient recipes.

Challenges

This was a project developed with other people, so strong communication and precise organisation were required. This was achieved through frequent meetings to discuss progress, alongside task tracking - and ultimately, was not a major problem.

The main challenge associated with this project was learning how to 3D model very quickly, and once again having to generalise. In this project, I had to design the game, produce all of the 3D assets, fully develop the potions/cauldron/orders systems and implement all game sound effects - within two weeks.

Fortunately, due to my increased experience with Unity, this project wasn't nearly as buggy initially, and was able to be produced - albeit with a reduction to the intended scope, within two weeks.

Database Explorer

Java Database Graphical

Summary

This project was written in Java using JSwing for the GUI. It is a graphical application designed for querying a database.

Information

This project was created in around 2 weeks, as a programming exercise and extension on and a rewrite of a university project

To create this project, I had to learn to create a GUI using Java - therefore I learned the basics of how to use JSwing.

Concept and Features

This project revolves around the formatting and searching of a database. To do this, I implemented my own SQL-like query language from scratch, which supports the use of several mathematical operators (<, <=, ==, >, >=), comparative string operators (begins, contains, equals), alongside logical operators (AND, OR, XOR, NAND, NOR, XNOR) and brackets to connect subqueries, alongside the NOT operator to invert a subquery. Subqueries will work both ways, and can be given in the form (DATA TYPE, OPERATOR, VALUE) or (VALUE, OPERATOR, DATA TYPE), with operations working as expected for all reversed subqueries.

To format the database, the database is scanned as it is read and data types are categorised as either numeric or a string (based on whether entries can be parsed as a float), and either filterable or non-filterable (based on how many unique entries there are). This is used to generate filters and choose the correct type of operator for the input given.

The language also supports the selection of any number of valid database data types in its arguments alongside the use of the wildcard (*) to select all data types. These can be chained in any order.

Alongside this, I built a class to combine any two given databases by a common type - the demo for this project uses a modified version of the publically availible IMDB database, having combined the ratings and basic databases.

Challenges

One of the major challenges associated with this project was learning how to use JSwing. JSwing uses a different form of UI creation to HTML - the only way to produce a complex UI is to use nested JPanels with varying layout managers, each focused on a hyperspecific form of layout. The difficulty in this was in various elements not working as expected or described, such as elements not rendering, positioning correctly, or recording their attributes correctly, necessitating workarounds.

The other major challenges was in producing the query parser and the necessary classes to flexibly and dynamically support query parsing. Setting up these classes was relatively complex compared to many of the projects I've worked on in the past, and this is the first time I've had to consider balancing cohesion and coupling in a solution.

Poker

Year

2024

Tags

Haskell Functional CLI-based

Summary

This project was written in pure Haskell. It is a CLI-based game designed to simulate games of Texas Hold Em' poker between both bots and humans.

Information

This project was created over around three weeks, and is a recreation of Texas Hold Em' poker, complete with support for AI bots and Jokers.

To create this project, I had to learn an entirely new programming paradigm - functional programming. Haskell is a pure functional language and thus works very differently from the procedural and object-oriented programming languages which I typically use.

Concept and Features

This game is a recreation of Texas Hold Em' poker, a popular card game.

In the version which I created, any number of human players can play alongside any combination of AI strategy - of which there are 4 implemented.
  • A random AI, which makes moves entirely based on chance without considering the game state.
  • A passive AI, an extension of the random AI with more passive tunings, which moves based on the current bet amount.
  • An aggressive AI, an extension of the random AI with more aggressive tunings, which moves based on its chip advantage.
  • A smart AI, which uses a complex function to evaluate its hand and how it should fare relatively, alongside a fuzzy logic-based decision tree to decide whether a bet is worth taking.
Additionally, Jokers were implemented as wildcards. Any number can be added to the deck, and will always produce the best possible result.

Challenges

This project necessitated learning an entierely new programming paradigm. Haskell, amongst many functional programming languages enforces a few strange practises, the most striking of which being the inherent limitation on the length of a function. The lack of a true for or while loop in the language forces a recursive solution to most problems, resulting in many functions and many function calls to accomplish what a single function can do in most other programming languages.

Other major complications included implementing an initially unplanned feature - Jokers. The addition of Jokers required a rewrite of all functions related to checking hand strength and getting a certain hand type. These rewrites typically significantly increased complexity as the logic to always get a Joker in the correct place is sometimes quite complex. One notably complex example was checking for straights - the function for this needs to prioritise making Jokers into as a high a card as possible while also producing a working straight every time, no matter where gaps in the straight may be.

Finally, the last major challenge was debugging in Haskell. Haskell is not only a pure functional programming language, but also evaluates expressions lazily, meaning that values are not calculated until they are required elsewhere in the program. This makes debugging far harder than it should be, as the debug trace will also not be evaluated unless the expression itself is, making debug behaviour unpredictable - for example, at one point in my project, I accidentally produced an infinitely recursive function and could not locate the cause. This was due to the infinite recursion occurring many lines down in a function from where the broken function was called.

Classifier

Year

2024

Tags

Python Classification Machine Learning

Summary

This project was written in Python using NumPy and SciPy. It is a classifier designed to classify images of numbers in the MNIST dataset.

Information

This project was created over a week, and is a classifier model built to identify the number represented by an image in the MNIST dataset.

Over the production of this project, I learned a large number of statistical techniques for classification, such as LDA, PCA, KL divergence, and k-NN classification.

Concept and Features

This classifier was built for the MNIST database, a popular database for training and testing machine learning and classification systems due to its relatively large size (60,000 training images, 10,000 testing images) and high data quality. It uses 28x28 greyscale images of numbers 0-9. I used 5,000 of these images with my classifier to increase speed.

My classifier was built to specifically identify noisy and masked MNIST data, and so features extensive dimensionality reduction - a technique used to simplify high-dimensional data, which in my case, after extensive parameter tuning, reduced the 784 parameters (28x28) down to 62 parameters. Because fewer, more important parameters are used, the system is much more resilient to noise, as noise typically has the greatest impact on empty pixels - which are usually unused as they are usually unimportant and portray little data, causing dimensionality reduction to discard them.

With my reduced parameters, I was able to achieve 95% accuracy on noisy data and 79% accuracy on masked data, which are relatively good results considering the reduced dataset that this model was trained on.

Challenges

This project necessitated learning a large amount of the mathematics used in image classification and translating it as efficiently as possible to Python. To help in handling large array, vector, and matrix operations, I learned the basics of NumPy, an efficient C-based Python library. Using the broadcasting feature of NumPy arrays allows functions to be quickly applied to all elements of an array, allowing for the model to be lazy-trained and classify 1,000 testing images within 5 seconds of it being run.

Additionally, there are many possible techniques for dimensionality reduction and classification, and it is hard to know which one is perfect for a given application. Initially, LDA - a technique used to reduce the dimensions of a dataset to a few dimensions which provide the most distinction between options - was used in conjunction with PCA, however this was found to reduce performance significantly, especially on masked data, and so was removed, leaving only PCA and KL divergence for dimensionality reduction. This amongst other techniques had to be gradually tweaked, such as classifier originally using Euclidean distance instead of cosine distance, which was also found to massively reduce performance on masked data.

Graphical Project

Summary

This project was written in Java using the JOGL library, a library which aims to import OpenGL functionality to Java. It is a basic animated scene which uses low-level graphical techniques and shaders.

Information

This is a graphical project built using the JOGL library for Java, alongside JSwing to handle UI elements. This was built over the course of 2 weeks as coursework for a university module.

This project is unique in that this was my introduction to GPU shaders.

Concept and Features

This project is an animated scene built using low-level graphics concepts. This consists of an animated bee, statues (which feature proximity detection), and a backdrop, made from billboarding trees and a sun. Each statue in the scene has a different affect on the bee, as shown in the demo - the wooden statue makes the bee flip over, the stone statue makes the bee circle around it, and the metal statue increases the flow of time. When a bee is in proximity of a statue, its face will change to reflect how long it has been near the bee.

Objects in the scene and their animations (which are mostly determined by simple sine wave functions) are handled using transforms applied to nodes in a scene graph, and rendered by the GPU using vertex and fragment shaders. The fragment shader supports multiple lights, multiple types of light (pictured are directional and spotlights, but point lights are also supported), and has partial support for transparency. It uses the Blinn-Phong reflection model to shade objects and supports light attenuation for point lights and spotlights.

Challenges

This is not my first graphical project, but is the first time I have worked in low-level graphics programming. The primary challenge of this project was expectedly learning shader code - shaders are highly parallel and require a different methodology behind their composition, as data is not directly transferrable between shader iterations. Instead, data is either provided by the previous shader (vertex shader to frag shader or frag shader to frag shader typically), or provided by the CPU, passed in as a uniform. Both of these methods work very differently to how typical CPU code handles data, and requires learning and planning to operate effectively.

Alongside this, all of my modelling in this project was done without the use of external tools like Blender, as I had used before in my Alchemy Game. This meant all models I produced had to be done with the use of a scene graph, using nodes to produce primitive shapes and the transforms which separated, scaled, and animated them.
